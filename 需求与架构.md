# 飞书待办提醒插件 — 需求与架构

## 一、核心需求

| 需求 | 描述 |
|------|------|
| **任务识别** | 用户在飞书私聊向小织发送文本或截图，LLM 自动提取任务名、描述、DDL、复杂度 |
| **任务创建** | 调用飞书 Task v2 API 创建待办，设置截止时间 + 原生 2h 提醒，指派给当前用户 |
| **分级提醒** | 复杂任务 DDL 前 2 天提醒，普通任务 DDL 前 1 天提醒，所有任务截止前 2h 提醒 |
| **口头自定义** | 用户可在输入时口头指定额外提醒时间（如"提前 3 天提醒我"） |
| **定时巡检** | 每 30 分钟扫描一次，匹配提醒规则后通过飞书 IM 主动推送 |
| **可视化查看** | 所有任务在飞书客户端原生「任务」面板中统一管理，无需额外页面 |
| **防重复提醒** | 每条规则只触发一次，标记后不再推送 |

## 二、权限需求

只需在飞书开放平台为应用添加两个 scope：

- `task:task:write` — 创建任务、设置提醒
- `task:task:read` — 查询任务列表

## 三、架构

```
飞书私聊（文本/截图）
        │
        ▼
  AstrBot → LLM (多模态)
        │
        │  Function Call
        ▼
┌─ 插件核心 ─────────────────────────┐
│                                    │
│  Function Tool:                    │
│    create_reminder_task(           │
│      summary, description,         │
│      deadline, is_complex,         │
│      custom_remind_days)           │
│         │                          │
│         ├── 飞书 Task API          │
│         │   创建任务 + 2h原生提醒   │
│         │                          │
│         ├── KV / 文件存储           │
│         │   持久化提醒规则 + umo   │
│         │                          │
│         └── 返回任务链接给用户      │
│                                    │
│  APScheduler / asyncio 定时任务:    │
│    每30分钟扫描已存任务             │
│    → 匹配规则 → 触发推送            │
│    → 标记已提醒                     │
│                                    │
│  主动推送: context.send_message()  │
│    通过 AstrBot 飞书适配器发 IM     │
└────────────────────────────────────┘
```

## 四、数据模型

```
task_{guid}: {
    summary:      string,
    deadline_ms:  int (UTC ms),
    is_complex:   bool,
    rules: [
        { minutes_before: int, triggered: bool },
        ...
    ],
    umo:          string (unified_msg_origin)
}

_task_ids: [guid, guid, ...]
```

## 五、开发注意事项

1. **不要用全局变量传递 Star 实例** — 应通过 AstrBot 提供的标准依赖注入或上下文传递机制，避免模块级单例和循环引用
2. **Function Tool 注册** — 使用 `self.context.add_llm_tools()` 在 `initialize()` 中注册，不要在 `__init__` 中注册
3. **异步全链路** — 所有 IO 操作（API 调用、存储、消息发送）必须是 async，使用 `httpx` / `aiohttp` 而非 `requests`
4. **时区处理** — 服务器/容器通常是 UTC，解析用户输入时需显式处理 `Asia/Shanghai` 偏移，避免巡检误判
5. **umo 的获取与使用** — `unified_msg_origin` 是主动推送的关键，需在工具回调中获取并持久化，确保巡检时能正确回推消息
6. **配置分离** — 敏感凭证（app_id / app_secret）存 `config.json`，放在 `data/plugin_data/` 下，不能混入插件代码目录
7. **依赖声明** — `requirements.txt` 中声明 `httpx`，AstrBot 重启时自动安装
8. **平台锁定** — `metadata.yaml` 中 `support_platforms: [lark]`，避免在 QQ/微信等平台误触发
9. **防抖与容错** — 巡检逻辑需包裹 try-except，单条任务失败不影响其他；API 调用需 token 缓存与过期刷新
10. **工具描述质量** — Function Tool 的 `description` 和 `parameters` 的 `description` 直接影响 LLM 提取精度，需用中文清晰描述每个字段的语义和判断标准
